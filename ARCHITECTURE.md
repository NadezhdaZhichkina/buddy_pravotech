# Архитектура Buddy — техническая документация

Подробное описание того, как работает Buddy с точки зрения инженера.

---

## 0. Два режима работы

| Режим | Файл | Описание |
|-------|------|----------|
| **Streamlit** | `streamlit_app.py` | Основной деплой на Streamlit Cloud. Чат, база знаний, самообучение, тикеты модерации. |
| **FastAPI + Mattermost** | `app/main.py` | Webhook для MChat. Опционально. |

См. **PROJECT_STRUCTURE.md** — краткий обзор структуры и env-переменных.

## 1. Обзор системы

Buddy — AI-агент онбординга. В режиме **Streamlit** (основной): друг новичка, отвечает из базы знаний, самообучается при «спасибо», направляет «что дальше» к менеджеру по адаптации.

В режиме **FastAPI** (Mattermost) Buddy:

1. **Встречает новичка** и узнаёт его роль
2. **Ведёт по сценарию онбординга** (шаги зависят от роли)
3. **Отвечает на вопросы** из базы знаний через LLM
4. **Переадресует неизвестные вопросы** коллегам в канал экспертов
5. **Сохраняет ответы коллег** в базу знаний и **передаёт их новичку**

## 3. Поток данных: как сообщение проходит через систему

### 3.1. Вход: Mattermost → Buddy

Mattermost при срабатывании исходящего webhook отправляет `POST` на `/mattermost/webhook`:

```json
{
  "user_name": "ivan.ivanov",
  "bot_user_id": "buddy-bot-id",
  "post": "{\"id\": \"abc123\", \"user_id\": \"user-xyz\", \"channel_id\": \"dm-channel-id\", \"message\": \"привет\", \"root_id\": \"\"}"
}
```

**Важно:**
- `post` — JSON-строка (Mattermost сериализует объект поста в строку)
- `channel_id` — ID канала, где написано сообщение (для DM — ID личного чата)
- `root_id` — если ответ в треде, ID корневого поста
- `bot_user_id` — ID бота; сообщения от бота игнорируются (защита от циклов)

### 3.2. Обработка в `handle_new_message`

```
                    ┌─────────────────────────────────────┐
                    │         handle_new_message           │
                    └─────────────────┬───────────────────┘
                                      │
                    ┌─────────────────▼───────────────────┐
                    │ post от бота? → return "ok" (игнор)  │
                    └─────────────────┬───────────────────┘
                                      │
                    ┌─────────────────▼───────────────────┐
                    │ Ответ в треде канала экспертов?      │
                    │ (root_id && channel_id == expert)    │
                    └─────────────────┬───────────────────┘
                         │ ДА                    │ НЕТ
                         ▼                       ▼
            ┌─────────────────────┐   ┌─────────────────────────────┐
            │ handle_human_answer │   │ get_or_create_user          │
            │ → сохранить ответ    │   │ → роутинг по user.state     │
            │ → отправить новичку │   └──────────────┬──────────────┘
            │ return "ok"         │                  │
            └─────────────────────┘                  ▼
                                    ┌─────────────────────────────────────┐
                                    │  state: new / start / привет / hi   │
                                    │  → awaiting_role                    │
                                    │  "Расскажи роль (backend, ...)"     │
                                    └─────────────────────────────────────┘
                                    ┌─────────────────────────────────────┐
                                    │  state: awaiting_role               │
                                    │  → onboarding, role = текст         │
                                    │  Первый шаг сценария                 │
                                    └─────────────────────────────────────┘
                                    ┌─────────────────────────────────────┐
                                    │  state: onboarding                  │
                                    │  → следующий шаг или qa             │
                                    └─────────────────────────────────────┘
                                    ┌─────────────────────────────────────┐
                                    │  state: qa                          │
                                    │  → handle_question()                │
                                    └─────────────────────────────────────┘
```

### 3.3. Выход: Buddy → Mattermost

**Исходящий webhook Mattermost** работает так: тело ответа HTTP (текст) Mattermost публикует как сообщение бота в том же канале, откуда пришёл запрос.

То есть `return "Привет! Я Buddy..."` в коде превращается в пост Buddy в DM новичка.

**Исключение:** когда эксперт отвечает в канале экспертов, мы возвращаем `"ok"` — Mattermost не должен дублировать это сообщением бота.

## 4. Модель состояний пользователя (User.state)

| Состояние       | Описание                                      | Следующее действие пользователя      |
|-----------------|-----------------------------------------------|--------------------------------------|
| `new`           | Первый контакт                                | Написать "привет" / "start" / "hi"   |
| `awaiting_role` | Ожидание роли                                 | Написать роль: backend, frontend…    |
| `onboarding`    | Идёт сценарий онбординга                      | Отправлять сообщения (шаг за шагом)  |
| `qa`            | Сценарий пройден, режим вопросов              | Задавать любые вопросы               |

Переход в `qa` происходит после последнего шага сценария. Дальше каждое сообщение считается вопросом.

## 5. Обработка вопроса (handle_question)

### 5.1. Поиск в базе знаний

Используется **ILIKE-поиск** по полям `question`, `answer`, `tags`:

```python
# Упрощённо
WHERE question ILIKE '%<текст вопроса>%'
   OR answer ILIKE '%<текст вопроса>%'
   OR tags ILIKE '%<текст вопроса>%'
ORDER BY created_at DESC
LIMIT 10
```

Результат — до 10 записей `KnowledgeItem`, которые передаются в LLM.

### 5.2. LLM (answer_from_knowledge)

1. **Контекст для LLM:** факты в формате `[Факт N] Вопрос: ... Ответ: ...`
2. **Системный промпт:** отвечать только по фактам; если ответа нет — явно сказать «нужно спросить у коллег»
3. **Определение результата:** по фразам «нужно спросить у коллег» / «не хватает информации» в ответе LLM возвращается `(known=False, answer)`

**Без OPENROUTER_API_KEY:**  
- если есть подходящие факты → `(True, "ответ есть, но LLM не настроен")`  
- если нет → `(False, "нужно спросить у коллег")`

### 5.3. Известный ответ

- Создаётся `Question` (status=ANSWERED) и `Answer` (author_type=bot)
- Текст ответа возвращается как тело HTTP → Mattermost показывает его в DM

### 5.4. Неизвестный ответ → эскалация экспертам

1. Создаётся `Question` (status=AWAITING_HUMAN, mattermost_channel_id=DM новичка)
2. `post_message()` — пост в канал экспертов: «Коллеги, вопрос от @user: … Ответьте в треде»
3. Сохраняется `mattermost_root_post_id` (ID этого поста)
4. Новичку возвращается: «Спросил у коллег, вернусь с ответом»

### 5.5. Ответ эксперта (handle_human_answer)

Срабатывает, когда приходит сообщение с `root_id` в канале экспертов:

1. По `root_id` находится `Question` (mattermost_root_post_id)
2. Создаётся `Answer` (author_type=human)
3. Создаётся `KnowledgeItem` (вопрос + ответ) — пополнение базы знаний
4. **Отправка новичку:** `post_message(channel_id=question.mattermost_channel_id, text=...)` — ответ уходит в DM новичка

## 6. Схема базы данных

```
┌──────────────────┐       ┌──────────────────┐       ┌──────────────────┐
│      users       │       │    questions     │       │     answers      │
├──────────────────┤       ├──────────────────┤       ├──────────────────┤
│ id (PK)          │───┐   │ id (PK)          │   ┌───│ id (PK)          │
│ mattermost_user  │   │   │ user_id (FK)     │◄──┤   │ question_id (FK)  │
│ username         │   └──►│ text             │   │   │ author_type      │
│ role             │       │ status           │   └───│ author_mm_user_id│
│ state            │       │ mattermost_root  │       │ text             │
│ onboarding_step  │       │ mattermost_channel│      │ created_at       │
│ created_at       │       │ created_at       │       └──────────────────┘
└──────────────────┘       └────────┬─────────┘
                                    │
                                    │ source_question_id (nullable)
                                    ▼
                    ┌──────────────────────────────┐
                    │      knowledge_items         │
                    ├──────────────────────────────┤
                    │ id (PK)                      │
                    │ question                     │
                    │ answer                       │
                    │ tags                         │
                    │ source_question_id (FK)      │
                    │ created_at                   │
                    └──────────────────────────────┘
```

**Важные поля:**
- `Question.mattermost_channel_id` — DM новичка, куда отправить ответ эксперта
- `Question.mattermost_root_post_id` — пост Buddy в канале экспертов, по `root_id` ответов в треде находим вопрос

## 9. Заполнение базы знаний

- **Скрипт:** `scripts/seed_knowledge.py`
- **Источник:** список `SEED_ITEMS` в коде (вопрос, ответ, теги)
- **Логика:** по `question` проверяется наличие записи; при отсутствии — вставка, при наличии — обновление `answer` и `tags` при изменении

## 10. Миграции БД

При старте приложения выполняется проверка:

- Есть ли в таблице `questions` колонка `mattermost_channel_id`
- Если нет — выполняется `ALTER TABLE questions ADD COLUMN mattermost_channel_id VARCHAR(64)`

Это нужно для обновления уже развёрнутых инстансов без пересоздания БД.

## 11. Потоки сценария онбординга

Сценарий задаётся в `app/onboarding.py`:

- **default** — общие шаги для всех
- **backend**, **frontend** — дополнительные шаги для ролей

`get_scenario_for_role(role)` возвращает `default + role_specific`. Для неизвестной роли — только `default`.

Каждое сообщение пользователя в состоянии `onboarding` сдвигает `onboarding_step_index` на 1. После последнего шага `state` переключается в `qa`.

| Текущее ограничение              | Возможное улучшение                          |
|----------------------------------|----------------------------------------------|
| ILIKE-поиск по подстроке         | Полнотекстовый поиск (FTS) или векторный     |

| Сценарий в коде                  | Хранение в БД или YAML                       |
| Нет уведомления о новых ответах  | Уже реализовано — ответ уходит в DM         |
| Нет метрик и логов               | Prometheus, структурированное логирование    |
